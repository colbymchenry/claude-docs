# Events System & Epic Auto-Close

## Events Table

The `events` table in each project's `.kommandr/kommandr.db` tracks issue activity for the "Recent Activity" feed.

### Schema
```sql
events (id, issue_id, event_type, actor, old_value, new_value, comment, created_at)
```

`event_type` is a plain text column — there is no enum or constraint. New types are added by convention, not migration.

### Existing Event Types
- `created` — new issue created
- `status_change` — status transition (old_value → new_value)
- `closed` — issue closed (old_value = previous status, new_value = 'closed')
- `comment` — added by `check-done-manager.ts` for system comments
- `comment_added`, `priority_change`, `assigned` — supported by UI but not yet auto-generated by stream polling

### Adding a New Event Type — Checklist

Three places must know about a new event type:

1. **`src/components/EventFeed.svelte`** — two spots:
   - `eventIcons` map (line ~7): add an icon name from the Icon component
   - `getEventDescription()` switch (line ~31): add a human-readable description

2. **Emit the event** — call `insertEvent()` from whichever code path triggers it:
   - Store interface: `src/lib/stores/project-store.ts:43`
   - SQLite impl: `src/lib/stores/sqlite-store.ts:344`
   - PostgreSQL impl: `src/lib/stores/postgresql-store.ts:502`
   - Legacy direct-DB: `src/lib/project-db.ts:483`
   - Proxy in `src/lib/stores/index.ts:148` delegates to the active store

   Signature: `insertEvent(issueId, eventType, actor, oldValue?, newValue?, comment?)`

3. **(Optional) Stream polling** in `src/routes/api/projects/[id]/stream/+server.ts` (~lines 138-147):
   - Auto-detects `created`, `closed`, and `status_change` by diffing issue state each poll cycle
   - If the new event corresponds to an Issue field change, add diff logic here
   - If triggered by an explicit API call, emit `insertEvent` directly in that route handler instead

No schema migration is needed for new event types.

### How Events Are Generated

Events are **not** written by the `updateIssue()` function directly. Instead, the **stream polling loop** in `src/routes/api/projects/[id]/stream/+server.ts` detects changes by comparing previous issue states to current states on each poll cycle (every 1 second). When it detects a status change or new issue, it calls `insertEvent()` from the project store.

This design is necessary because external writes (e.g., from MCP tools) may bypass the dashboard's `updateIssue()` function.

### Key Functions
- `insertEvent(issueId, eventType, actor, oldValue?, newValue?, comment?)` — writes to `events` table
- `getRecentEvents(since?, limit?)` — reads events for the activity feed
- `autoCloseEpicIfComplete(epicId)` — checks all children; closes epic if all are closed

## Epic Auto-Close

When the stream detects a task moving to `closed` status, it checks if the task has a parent epic (via `getParentIssue()`). If the parent is an epic and all its children are now closed, `autoCloseEpicIfComplete()` automatically:
1. Updates the epic's status to `closed` and sets `closed_at`
2. Inserts `status_change` and `closed` events for the epic

### Data Flow
```
Kommandr writes to SQLite → stream polls getDataVersion() → detects change
→ compares previousIssueStates map → generates events via insertEvent()
→ checks parent epics via autoCloseEpicIfComplete()
→ sends updated issues + events via SSE → EventFeed.svelte renders them
```

## Frontend

- `EventFeed.svelte` — renders up to 20 events with icons and relative timestamps
- `project-stream.svelte.ts` — accumulates events from SSE `update` messages, keeps rolling list of 100
- Events are passed as `stream.events` from the project page to `EventFeed`
